EmbLink UI/ designer's guide
Design system for EmbLinkOS

Beautiful interfaces, composed — not decorated.

EmbLink UI is a from-scratch toolkit that renders every pixel itself: SDF-antialiased shapes, a runtime TrueType rasterizer, flexbox layout, and fine-grained reactivity. You describe a screen by composing widgets; a coherent, modern look falls out of the design tokens automatically. You never pick a raw colour, size, or font.

Rendered by EmbLinkOS, not a mockup. Both frames above are the exact output of the toolkit drawing the same 40-line screen — the only difference is one call: ui_theme_use_dark(true).

Read once

The model in a minute

Five ideas carry the whole system. If these click, everything else is just widget names.

01

Declare every frame

You describe what should be on screen right now, top to bottom, by calling widgets. Re-declaring the same UI is free — the toolkit diffs it and does zero work. There is no "create" or "update" step.

02

State is plain data

Your state lives in ordinary variables (or signals, for reactive updates). The UI is a pure function of that state — change the data, the next frame reflects it.

03

Controls return events

A button returns true the frame it's clicked; a toggle returns true when tapped. You react by mutating your state — no callbacks, no wiring.

04

Layout is flexbox

Rows and columns with spacing, padding, alignment, and flexible spacers. Sizes are FIXED, INTRINSIC (fit content), or FLEX (grow / shrink to share space).

05

Everything comes from the theme

Colour, spacing, radius, type, and elevation are all semantic tokens. Widgets read them so your screen is coherent by construction — and the whole OS restyles, light to dark, with a single call.

Building blocks

The widget kit

Compose these. Each reads the tokens, so spacing, colour and elevation are consistent without a single hard-coded value.

ui_screen_begin / _endThe page. Themed background, standard page padding, a vertical stack.
ui_card_begin / _endAn elevated surface — hairline border, soft shadow, comfortable padding. The default container for content.
ui_panel_begin / _endA quieter inset surface for grouping (settings sections, wells).
ui_row_begin / _endA horizontal stack, vertically centred, with standard gap.
ui_col_begin / _endA vertical stack with standard gap.
ui_heading / ui_titleBold type for section and screen headers.
ui_body / ui_secondary / ui_captionThe three-step text hierarchy — primary, supporting, and metadata.
ui_button_primaryThe one confident action per view. Accent fill, returns true on click.
ui_button_secondary / _ghostLower-emphasis actions — bordered, or text-only.
ui_toggle(bool)An on/off switch. Returns true when tapped; you flip your own state.
ui_badge(text, tone)A small status pill — accent, success, warning, danger, or neutral.
ui_dividerA full-width hairline rule between groups.
ui_flex_spacer / ui_gapPush siblings apart (flexible), or insert a fixed gap.
Arranging things

Layout, the flexbox way

A row or a column arranges its children along a main axis. Each child's size on that axis is one of three modes:

FIXED

An exact size in pixels. Use sparingly — for things with an intrinsic footprint like an avatar or a toggle.

INTRINSIC

Fit the content. A label is as wide as its text; a button as wide as its padding + label. The default.

FLEX

Grow to fill leftover space (weight flex_grow) or shrink when space is tight (flex_shrink, CSS-accurate). A spacer is just flex_grow:1.

Alignment controls the cross axis (ALIGN_START / CENTER / END / STRETCH); justify distributes the main axis (START / CENTER / END / SPACE_BETWEEN). The everyday idiom — label on the left, control on the right — is just a row with a ui_flex_spacer() between them:

ui_row_begin(0);
    ui_body("Notifications");
    ui_flex_spacer();                      // eats the middle
    if (ui_toggle(state.on)) state.on = !state.on;
ui_row_end();
Putting it together

A complete screen

This is the entire source behind the card at the top of this page. Note there are no colours, sizes, or fonts in it — only structure and state.

void app(void) {
    ui_screen_begin();                          // themed page
    ui_set_justify(JUSTIFY_CENTER);
    ui_set_align(ALIGN_CENTER);                  // centre the card

    ui_card_begin(1);
        ui_row_begin(0);
            ui_title("Display");
            ui_flex_spacer();
            ui_badge("Preview", BADGE_ACCENT);
        ui_row_end();
        ui_secondary("Adjust how EmbLink looks on this screen.");

        ui_divider();

        ui_row_begin(0);
            ui_body("Dark mode");
            ui_flex_spacer();
            if (ui_toggle(dark)) dark = !dark;
        ui_row_end();

        ui_divider();

        ui_row_begin(0);
            ui_flex_spacer();
            ui_button_secondary("Cancel");
            if (ui_button_primary("Save changes")) save();
        ui_row_end();
    ui_card_end();

    ui_screen_end();
}

Start from ui/template/app_template.c — copy it, rename app(), and build inside. A runnable host example that renders app() to a PNG lives in ui/showcase/showcase.c.

How it runs

The frame loop

On EmbLinkOS a UI process runs one loop. You rarely write it by hand — but knowing the shape explains why re-declaring every frame is cheap.

// once per frame, for a client window:
1  drain input messages    // clicks/keys -> your handlers call signal_set()
2  reactivity_flush()      // re-run only the scopes whose signals changed
3  app() + ui_run_layout()  // declare UI, then flex-solve geometry
4  scene_render_frame()    // paint only the dirty rectangles
5  surface_commit()        // hand the buffer to the compositor
6  wait for FRAME_DONE     // pace to the compositor, then loop
Why identical redeclares are free. Step 3 diffs your declaration against the retained tree and skips any property that didn't change; step 4 repaints only the pixels that actually moved. A screen that declares the same thing twice does no layout and no paint work at all — so you're free to describe the whole UI every frame without worrying about cost.
Taste

Principles for a clean screen

Hierarchy by size, weight & colour

Not by boxes and rules. A bold title, a body line, a tertiary caption — three steps are almost always enough.

One accent, spent once

The primary button, a selected state, a key link. If two things compete for the accent, neither wins. Everything else stays neutral.

Rhythm from the grid

Reach for the spacing scale, not arbitrary numbers. Consistent gaps read as calm; mixed ones read as noise.

Status colour ≠ accent

Success, warning, danger carry meaning — keep them separate from the accent so "this needs attention" reads instantly.
